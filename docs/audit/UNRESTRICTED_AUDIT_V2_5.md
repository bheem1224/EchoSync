# EchoSync v2.5.0 "Total Freedom" Unrestricted Audit

## Risk Assessment Matrix

| Severity | Location | Description | Conceptual Remediation |
|---|---|---|---|
| **CRITICAL** | `services/metadata_enhancer.py` (line 453-568) | **Transaction Deadlock**: Long-running external API calls (`search_by_isrc`, `resolve_fingerprint_details`, `search`) are executed inside the `db.session_scope()` block. A slow network client or rate-limiting delays from MusicBrainz/AcoustID will hold the SQLite database lock open for extended periods, causing transaction deadlocks across the entire application. | Move all external network lookups outside the database transaction block. Fetch data first, then open a brief session to persist the updates. |
| **CRITICAL** | `core/plugin_loader.py` (AST Scanner) | **Plugin Sandbox Escape (Reflection via Exceptions)**: The AST scanner blocks direct calls to `__builtins__`, `getattr`, `importlib` and blocks reflection attributes like `__class__`. However, an attacker can throw and catch an exception, access the traceback (`e.__traceback__.tb_frame`), and traverse up the call stack (`frame.f_back`) to extract `frame.f_globals['__builtins__']`. This allows arbitrary OS command execution (`__import__('os').system(...)`), completely bypassing the Zero-Trust Sandbox. | Block the `__traceback__` attribute in the `PluginSecurityScanner` (`visit_Attribute`), block the `sys` module comprehensively, and ideally run community plugins in a severely restricted `RestrictedPython` environment or sub-process rather than relying solely on AST blocklists. |
| **HIGH** | `web/routes/metadata_review.py` (line 442) | **Unbound Path Traversal / LFI via Stream API**: The `GET /api/review-queue/<id>/stream` endpoint extracts `task.file_path` from the database, expands user paths, and resolves it strictly. However, it fails to perform any verification against `jail.py`'s allowed roots (`.is_relative_to()`). If an attacker or malicious plugin inserts a `ReviewTask` with `file_path = "/etc/passwd"` (e.g. via the `/queue/approve` API which lacks path validation on `metadata`), the server will blindly return the file contents. | Enforce `file_jail.validate(file_path)` inside `_resolve_task_file` before returning the file path. |
| **HIGH** | `core/plugin_store.py` (line 122) | **Zip Slip / Partial Path Match Vulnerability**: During plugin installation, the server attempts to prevent Zip Slip directory traversal by ensuring the target extraction path falls inside the plugin directory: `if not str(target_file).startswith(str(dest_dir.resolve())):`. However, string `startswith()` on paths is flawed when `dest_dir` lacks a trailing slash. An attacker can craft a zip file containing `../plugin-malicious/pwned.py`. If `dest_dir` is `/app/plugins/plugin`, the target becomes `/app/plugins/plugin-malicious/pwned.py`. Since it still starts with the substring `/app/plugins/plugin`, the check passes, allowing files to be extracted outside the designated plugin folder. | Replace string `startswith()` with the robust `target_file.is_relative_to(dest_dir.resolve())`. |
| **MEDIUM** | `core/plugin_store.py` (line 166) | **SQL Injection in Plugin Uninstallation**: When dropping database tables associated with a plugin, the uninstaller builds the query using `f"SELECT name FROM sqlite_master WHERE type='table' AND name LIKE '{prefix}'"`. Although it creates a `safe_id`, the underlying `plugin_id` is substituted into the `LIKE` clause as a string interpolation without parameterized queries. While the attacker requires prior control of the plugin ID being passed to the endpoint, this violates secure DB practices. | Use parameterized queries for `execute()` rather than string interpolation for the `LIKE` clause: `conn.execute(text("... LIKE :prefix"), {"prefix": prefix})`. |
| **MEDIUM** | `core/webhook_parsers.py` (line 10) | **DNS Rebinding TOCTOU in SSRF Protection**: The `validate_safe_url` function prevents Webhook SSRF by resolving the URL's hostname via `socket.gethostbyname` and checking if the IP is private/loopback. If an attacker controls a domain, they can configure it to resolve to a safe IP during validation, then rapidly rebind the DNS record to an internal IP (e.g., `127.0.0.1:8080`) immediately after, leading to potential SSRF if the system subsequently fetches the URL. | Implement an HTTP client configuration that strictly binds requests to the IP resolved during the validation phase rather than allowing the HTTP library to re-resolve the hostname, or configure network-level egress filtering. |
