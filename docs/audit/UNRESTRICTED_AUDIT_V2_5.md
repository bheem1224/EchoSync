# Unrestricted Security & Architecture Audit - EchoSync V2.5.0

## 1. Risk Assessment Matrix

| Vulnerability | Severity | Location | Description | Conceptual Remediation |
|---|---|---|---|---|
| IDOR / Broken Access Control | HIGH | `web/routes/auth.py` and across `web/routes/` | The API lacks any global authentication checks. Most routes in `web/routes/` do not verify tokens, sessions, or user identity. A user could hit any API directly, modify jobs, delete history, etc. `auth.py` stubs `/status` with `{"authenticated": False}` but other endpoints don't enforce authentication in standard paths. | Implement a `@require_auth` decorator or middleware on all private API endpoints. Ensure operations acting on user data (like PlaybackHistory) cryptographically verify the user ID belongs to the token holder. |
| Potential SSRF / Webhook Abuse | MEDIUM | `web/routes/webhooks.py` -> `core/webhook_parsers.py` | The `/api/webhooks/<provider>` endpoint blindly accepts payloads from `request.form` or `request.get_json`. Although it restricts dispatch to `PlexWebhookParser` or `NavidromeWebhookParser`, if `metadata.ratingKey` or `metadata.guid` can be forced to an internal URI or unvalidated format by a malicious user via their media server, it could poison the local database state or create malformed sync IDs (though direct internal network requests from the app itself appear somewhat mitigated as it merely saves to the DB). | Ensure the `provider_item_id` and parsed `sync_id` strictly conform to expected regex/URN schemas (e.g. `ss:track:mbid:*`) before interacting with the database or triggering downstream processing. |
| Zip Bomb during Plugin Install | MEDIUM | `core/plugin_store.py` (`download_plugin`) | The zip extraction routine correctly checks for Zip Slip (path traversal) but does not limit the total uncompressed size or the compression ratio of the extracted files. A malicious plugin could be a zip bomb causing memory/disk exhaustion or DoS when extracted using `z.read(zi)` and `f.write()`. | Implement a cumulative uncompressed size counter during extraction and abort if it exceeds a reasonable threshold (e.g., 50MB) or a maximum compression ratio. |
| SQL Injection (Plugin Uninstall) | HIGH | `core/plugin_store.py` (`uninstall_plugin`) | The uninstall logic loops through database tables dropping those matching `prefix = f"plugin_{safe_id}_%"`. However, in the loop `for (table_name,) in tables:`, it executes `conn.execute(text(f"DROP TABLE IF EXISTS {table_name}"))`. If a malicious plugin manages to create a table with a crafted name (which is generally restricted but potentially possible via zero-trust bypass or other flaws), it could result in SQL injection since table names aren't properly quoted using SQLAlchemy metadata constructs or basic parameterized safeguards where possible. | Do not use f-strings directly inside `conn.execute(text())`. Since parameters can't be used for identifiers like table names in SQLite, properly escape the identifier by wrapping it in double quotes (e.g. `\"\"\"{table_name.replace('\"', '\"\"')}\"\"\"`). |
| Transaction Deadlocks / Lock Holding | HIGH | `web/routes/metadata_review.py` (`lookup_review_queue_item_acoustid`, `lookup_review_queue_item_musicbrainz`) | The API endpoints for AcoustID and MusicBrainz metadata lookups perform external HTTP calls (via the providers) while holding an active SQLite database session block (`with db.session_scope() as session:`). This causes the SQLite connection/lock to be held open for the entire duration of the external API request. In concurrent use, this will easily lead to SQLite transaction deadlocks or `database is locked` errors. | Only open the database session to fetch the task data, close the session, perform the slow external network lookup, and then open a new session to update the task state. |
| Sandbox Escape (GC/Reflection) | HIGH | `core/plugin_loader.py` (`PluginSecurityScanner`) | The AST scanner explicitly blocks `os`, `subprocess`, `sqlite3`, `sys`, `importlib`, `database`, `inspect`, and `ctypes` as well as several internal `__` methods. However, it fails to block the `gc` (garbage collector) module. An attacker could import `gc`, run `gc.get_objects()`, and find references to imported modules (like `os` or `sqlite3`) or active database connections that were loaded by the main application, effectively bypassing the AST `import` restrictions and breaking out of the sandbox. | Add `gc` to the list of forbidden imports in `visit_Import` and `visit_ImportFrom`. |
| Broken Module Load Handling | LOW | `core/plugin_loader.py` (`_load_provider_package`) | If a plugin fails to load due to a syntax error or a missing import, the fallback catch-all uses an unsafe logic pattern: `config_manager.disable_provider(f"plugin.{name}")` which correctly disables it, but the exception might leave partial or broken registry entries depending on when the failure occurs during the import sequence. Additionally, `get_provider_capabilities` throws a fatal `KeyError` if a provider is passed that is not in the registry instead of returning a default or None, leading to 500 errors if a disabled provider is requested. | Ensure `get_provider_capabilities` defaults gracefully, and clean up the ProviderRegistry upon import failure. |
| Plugin Database Escapes | LOW | `database/working_database.py` (`ProviderStorageBox`) | The `ProviderStorageBox` ensures dynamic tables are prefixed with `prv_{provider_name}_` during creation. However, the plugin still gets back a raw SQLAlchemy `Table` object and executes against `self.engine`. There is nothing strictly preventing the plugin from bypassing `ProviderStorageBox` entirely to use `db.engine.execute()` directly to mutate or drop core tables like `users` or `tracks` if it ignores the sandbox wrapper. | Pass a restricted proxy or session to the plugin instead of exposing the raw database engine, blocking DDL queries (e.g., DROP TABLE) on non-prefixed tables. |
| Plugin Memory Leaks (Event Bus) | LOW | `core/event_bus.py` (`EventBus.subscribe`) | The EventBus `subscribe` method does not provide a mechanism to unsubscribe. If a plugin registers a listener and is later reloaded or fails, the handler remains registered indefinitely, leading to memory leaks and potential runtime errors if the handler relies on an invalidated plugin context. | Implement an `unsubscribe` method in `EventBus` and ensure `PluginLoader` or plugin uninstallation routines clean up event listeners belonging to the unloaded plugin. |
| Missing Plugin Graceful Degradation | LOW | `services/metadata_enhancer.py` | If the system has no active `FETCH_METADATA` provider (e.g., the MusicBrainz core plugin is disabled or errors out), `self._get_provider(Capability.FETCH_METADATA)` returns `None`. The enhancer methods mostly check `if metadata_provider:` but there are edge cases (like `identify_file_batch`) where `metadata_provider` is assumed to exist when trying to use `getattr(metadata_provider, "get_release", None)`. An exception here would derail the batch identification. | Add strict `None` checks whenever invoking methods on providers fetched dynamically from `get_provider()`. |
