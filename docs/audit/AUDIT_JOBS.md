# AUDIT_JOBS.md (Scheduling and Concurrency Audit)

## 🚨 CRITICAL BLOCKERS

- [ ] **Concurrency Lock Leaks (`core/job_queue.py`):** In `JobQueue.execute_job_now` (line 240) and `JobQueue._execute_job` (line 363), the mutex lock `self._is_running[name] = True` is set *outside* of the `try...finally` block that wraps the background thread execution. If `threading.Thread(target=_run_job_thread).start()` or `threading.Thread(target=worker).start()` fails to spawn (e.g., due to system resource limits), the `finally` block containing `_finalize_job_after_run` will never execute. This permanently leaks the lock, trapping the job in a perpetual "running" state and blocking future executions until restart.
- [ ] **The 'Pending' Initialization Bug (`core/job_queue.py`):** In `JobQueue.register_job` (line 135), the initial `next_run` timestamp is calculated as `time.time() + max(start_after, 0.0)`. The `start_after` argument defaults to `0.0`, meaning all registered jobs schedule to run immediately (`time.time()`) upon startup. There is no fallback logic to offset `next_run` by `interval_seconds` if `start_after` is omitted. If the initial run is delayed or misses its execution window, the job appears stuck in 'Pending' because it lacks a correctly calculated future interval timestamp.
- [ ] **Rogue Thread: Download Manager (`services/download_manager.py`):** A standalone thread is spawned manually in `DownloadManager.start_background_task` (line 288) using `threading.Thread(target=_runner, daemon=True)`. This bypasses the central registry (`core/job_queue.py`) and makes it untrackable by the system's job monitoring.
- [ ] **Rogue Thread: Tidal OAuth Sidecar (`providers/tidal/client.py`):** The Tidal API client starts a manual thread for its callback server in `TidalClient._start_callback_server` (line 388) using `threading.Thread(target=self.auth_server.serve_forever)`. This standalone scheduler should be standardized or evaluated against the central OAuth sidecar.
- [ ] **Rogue Timer: Plex OAuth Polling (`providers/plex/routes.py`):** `plex_auth_status` starts a manual background cleanup thread (line 230) containing a hardcoded `time.sleep(900)` blocking delay (line 224) instead of utilizing an APScheduler/job queue delayed execution task.
- [ ] **Rogue Thread: Database Update Worker (`core/database_update_worker.py`):** The system relies on a standalone worker class inheriting directly from `threading.Thread` (or `QThread`), initiated on line 20. While invoked by `core/system_jobs.py`, its heavy inner state manages its own asynchronous thread lifecycle instead of fully delegating execution to the `JobQueue` worker pool.
- [ ] **Rogue `while True` loop: Jellyfin Client (`providers/jellyfin/client.py`):** Manual infinite polling loops exist on lines 559 and 622, completely detaching this background activity from the central queue's graceful shutdown and error-handling mechanics.
- [ ] **Rogue `while True` loop: Tidal API v2 (`providers/tidal/api_v2.py`):** Manual infinite polling loops exist on lines 37 and 62, detaching this execution from the central queue constraints.

## Warnings / Tech Debt

- [ ] **Missing Registration (`core/system_jobs.py`):** The job `user_history_sync` is completely missing from `register_all_system_jobs()`. It must be registered with an interval of `43200` seconds (12 hours).

While expected for bootstraps, they act outside of the unified job scheduler boundaries.
