## 1. Error Swallowing & State Desync

### 1.1 ‚úÖ Bare `except: pass` on provider init ‚ÄĒ `services/sync_service.py` lines 124, 129, 134
Three consecutive `try/except: pass` blocks silently swallowed any failure when lazy-loading the Plex, Jellyfin, and Navidrome clients. A misconfigured provider would leave the client as `None` and cause a confusing `AttributeError` downstream rather than a clear root-cause log.  
**Fix:** Replaced with `except Exception as e: logger.warning(...)` on all three blocks.

### 1.2 ‚úÖ Bare `except:` in slskd `authenticate()` ‚ÄĒ `providers/slskd/client.py` ~line 800
A completely bare `except:` (catching `BaseException`, including `KeyboardInterrupt` and `SystemExit`) silently returned `False`.  
**Fix:** Replaced with `except Exception as e: logger.debug(...)`.

### 1.3 ‚ö†ÔłŹ `except Exception: pass` in job retry path ‚ÄĒ `core/job_queue.py` lines 238‚Äď245
If the job function succeeds but a subsequent status-update step raises, the retry counter is not reset and the job is marked as a failure even though the core work completed.  
**Not fixed:** Requires a considered retry-state redesign; a naive patch could cause double-execution.

## 2. Concurrency, Async & Threading

### 2.1 ‚úÖ Non-thread-safe singleton ‚ÄĒ `services/download_manager.py` line 56
```python
# Before
if cls._instance is None:
    cls._instance = DownloadManager()   # race: two threads can both pass the check
```
Two Flask worker threads arriving simultaneously could each construct a `DownloadManager`, producing two independent queues and two background loops.  
**Fix:** Double-checked locking with a class-level `threading.Lock`.

### 2.2 ‚úÖ Non-thread-safe singleton ‚ÄĒ `services/auto_importer.py` line 42
Same pattern as 2.1.  
**Fix:** Same double-checked locking.

### 2.3 ‚úÖ Non-thread-safe module singleton ‚ÄĒ `core/caching/provider_cache.py` line 204
Module-level `get_cache()` used an unprotected `if _cache_instance is None` check.  
**Fix:** Added `_cache_lock = threading.Lock()` and double-checked locking.

### 2.4 ‚úÖ `EventBus._subscribers` not initialised in `__init__` and read/written without a lock ‚ÄĒ `core/event_bus.py`
`_subscribers` was initialised lazily with `if not hasattr(self, "_subscribers")` inside `subscribe()` and `publish_lightweight()`. Concurrent calls could create the dict twice or read a partially-written copy.  
**Fix:** Initialised in `__init__`; all reads and writes now copy under `self._lock`.

### 2.5 ‚úÖ `RateLimiter.wait()` post-`await` window overshoot ‚ÄĒ `core/rate_limiter.py` lines 40‚Äď60
After sleeping for the rate-limit window, the method appended a timestamp without re-checking whether another concurrent coroutine had already consumed the available slot during the sleep.  
**Fix:** Replaced the single-check `if` with a `while` loop that re-cleans and re-checks after every sleep.

### 2.6 ‚úÖ Global `_last_call` dict written without a lock ‚ÄĒ `core/rate_limiter.py` lines 11‚Äď27
The `@rate_limited` decorator's backing dict was read and written from multiple threads without synchronisation.  
**Fix:** Added `_last_call_lock = threading.Lock()` wrapping both the read and the write inside `wrapper()`.

### 2.7 ‚úÖ `asyncio.set_event_loop(loop)` inside sync wrappers ‚ÄĒ `providers/slskd/client.py` lines 744‚Äď800
Each of the four sync wrappers (`search`, `download`, `get_download_status`, `authenticate`) called `asyncio.set_event_loop(loop)` before running the coroutine, and then closed `loop`. After `loop.close()` the thread's event-loop pointer was left pointing at a **dead** loop, corrupting any later asyncio usage on that thread.  
**Fix:** Removed all four `asyncio.set_event_loop(loop)` calls. The new loop is created, used, and closed without touching the thread's global loop pointer.

### 2.8 ‚ö†ÔłŹ `asyncio.Semaphore` created in `__init__` ‚ÄĒ `providers/slskd/client.py` line 186
In Python ‚Č§ 3.9 an asyncio primitive created outside a running loop would be attached to a loop that might not be the one it was later `await`-ed from. **Not a bug on Python 3.10+** (loop attachment was removed from primitives). The project runs on CPython 3.14, so this is safe as-is.

### 2.9 ‚ö†ÔłŹ Daemon threads without tracked shutdown ‚ÄĒ `core/database_update_worker.py`, `core/job_queue.py`, `providers/tidal/client.py`
Several daemon threads are started with no way to join or signal them gracefully (beyond process exit). Low risk because daemon threads are killed automatically on interpreter shutdown; fixing them would require a broader lifecycle refactor.

### 2.10 ‚ö†ÔłŹ Lock held during callback invocation ‚ÄĒ `core/media_scan_manager.py` lines 97‚Äď135
`_call_completion_callbacks()` is invoked from within `_lock`-holding code paths. If a callback itself calls back into the manager (e.g. `request_scan()`), it will attempt to acquire `_lock` and deadlock. Currently no registered callback does this, but it is a fragile pattern. The existing `_call_completion_callbacks` helper already copies the list before iterating, which prevents iterator-invalidation; a full fix would require releasing the lock before dispatching.

## 3. Memory Leaks & Object Lifecycles

### 3.1 ‚úÖ `AutoImportService._recently_completed` dict unbounded growth ‚ÄĒ `services/auto_importer.py` lines 36‚Äď37
Old entries were only removed when the **same file was re-encountered**. Unique files processed once stayed in the dict forever. A service running continuously for weeks would accumulate thousands of stale entries.  
**Fix:** Added a bulk-prune step at the start of every `process_batch()` call that evicts all entries older than 10 seconds in one pass.

### 3.2 ‚úÖ `MediaScanManager._scan_completion_callbacks` never cleared on shutdown ‚ÄĒ `core/media_scan_manager.py` line 356
`shutdown()` cancelled both timers but left the callback list intact. If the manager was re-initialised (e.g. after a config reload), the stale callbacks would fire on the very next scan cycle.  
**Fix:** Added `self._scan_completion_callbacks.clear()` inside `shutdown()`.

### 3.3 ‚ö†ÔłŹ `DownloadManager._active_providers` cache has no config-change invalidation ‚ÄĒ `services/download_manager.py`
Provider instances are cached on first use (the fix applied earlier in this session). If the user changes provider credentials or disables a provider at runtime, the cached instance will continue to be used until the process restarts.  
**Not fixed:** Requires a cache-invalidation hook tied to config writes. Added as a known limitation.

### 3.4 ‚ö†ÔłŹ `RequestManager` HTTP session never explicitly closed ‚ÄĒ `core/request_manager.py` lines 67‚Äď70
`requests.Session` objects are created in `__init__` but rely on garbage collection to release the underlying connection pool. With CPython's reference counting this is usually prompt, but it is not guaranteed and can leave sockets open under PyPy or in circular-reference cases.

### 3.5 ‚ö†ÔłŹ Jellyfin library caches have no TTL or eviction ‚ÄĒ `providers/jellyfin/client.py` lines 283‚Äď291
`_album_cache`, `_track_cache`, and `_artist_cache` grow without bound for large libraries and are never reset unless the client is re-instantiated. `_cache_populated` is set to `True` once and never cleared, so cache data can go stale after server-side library changes.

## 4. Performance & Query Optimisation

### 4.1 ‚ö†ÔłŹ Unbounded `.all()` on download queue ‚ÄĒ `web/routes/downloads.py` line 36
`session.query(Download).all()` loads the entire download table. As the queue grows this consumes unbounded RAM and makes the UI increasingly slow.  
**Recommendation:** Add `.order_by(...).limit(200).offset(page * 200)` and wire up pagination in the front-end.

### 4.2 ‚ö†ÔłŹ Unbounded `.all()` on pending review tasks ‚ÄĒ `web/routes/metadata_review.py` line 286 and `web/routes/metadata.py` line 77
Both load all pending `ReviewTask` rows with no page limit.  
**Recommendation:** Same pagination pattern.

### 4.3 ‚ö†ÔłŹ Multiple unbounded `.all()` in manager dashboard ‚ÄĒ `web/routes/manager.py` lines 196‚Äď210, 274‚Äď290, 502‚Äď510
`UserTrackState`, `UserRating`, and rating-distribution queries all use `.all()` with no cap.

### 4.4 ‚ö†ÔłŹ N+1 query in waterfall search ‚ÄĒ `services/download_manager.py` lines 335‚Äď353
Items are fetched in a batch query (`.limit(30).all()`), but inside `_execute_waterfall_search_and_download` each item is retrieved again individually via `session.query(Download).get(download_id)`. For 30 concurrent search tasks that is 30 individual queries instead of 1. The per-task isolation is intentional (fresh state after potential concurrent updates), so fixing this cleanly would require loading only the immutable fields (the serialised `soul_sync_track` JSON) in the batch and using the individual reload only for status checks.

### 4.5 ‚ö†ÔłŹ N+1 query in duplicate analysis ‚ÄĒ `services/library_hygiene.py` lines 44‚Äď56
For each fingerprint-hash group a separate `session.query(Track)` is issued. Could be replaced with a single query using `WHERE fingerprint_hash IN (...)`.
